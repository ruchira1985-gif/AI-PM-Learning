# PRD Skill - How to Write a Product Requirements Document

> A guide for product managers, developers, and stakeholders to create effective PRDs.

---

## What is a PRD?

A **Product Requirements Document (PRD)** is a living document that defines:
- What product/feature to build
- Why it's being built
- Who it's for
- How it should work

---

## When to Write a PRD

✅ **Write a PRD when:**
- Building a new feature
- Making significant changes to existing functionality
- Starting a new project
- Onboarding new team members

❌ **Skip the PRD when:**
- Small bug fixes
- Minor UI tweaks
- Quick experiments

---

## PRD Structure

### 1. Overview Section
- **One-liner:** What is this in one sentence?
- **Problem Statement:** What user pain point does it solve?
- **Success Criteria:** How do we know it worked?

### 2. Features List
- Use checkboxes for tracking
- Distinguish between "must-have" vs "nice-to-have"
- Include acceptance criteria for each

### 3. User Flow
- Walk through the happy path
- Include error paths
- Map edge cases

### 4. Technical Specs
- Be specific but not over-prescriptive
- Include constraints and assumptions
- Document dependencies

---

## Writing Tips

### Do ✅
- Keep it concise and scannable
- Use clear, simple language
- Include visuals (mockups, diagrams)
- Update as requirements change
- Get sign-off from stakeholders

### Don't ❌
- Write paragraphs instead of bullet points
- Include implementation details (that's for engineering)
- Lock it away - make it accessible
- Forget to review and update

---

## Common Mistakes

| Mistake | Why It's Bad | Fix |
|---------|--------------|-----|
| Too vague | Team can't execute | Add specific acceptance criteria |
| Too detailed | Becomes a spec, not a requirement | Focus on "what", not "how" |
| No user research | Building in a vacuum | Include user data/feedback |
| No success metrics | Can't measure impact | Define KPIs upfront |
| Out of scope creep | Project bloat | Clearly list what's NOT included |

---

## PRD Review Checklist

Before sharing your PRD, verify:

- [ ] Clear problem statement
- [ ] Target users defined
- [ ] All features have acceptance criteria
- [ ] User flow is documented
- [ ] Technical requirements listed
- [ ] Non-functional requirements covered
- [ ] Success metrics defined
- [ ] Timeline/milestones included
- [ ] Risks identified
- [ ] Out of scope clearly stated
- [ ] Stakeholders have reviewed
